Задача 2.
- Сделать верхнеуровневую постановку в формате User story и описать требования к новой функциональности в формате 
вариантов использования (Use Case).

User story:
- Как продавец;
- Хочу опубликовать свой товар на витрине в маркетплейсе;
- Чтобы привлечь внимание покупателей к своему товару, начать продавать его и увеличить выручку.
Критерии приемки:
- Можно заносить всю необходимую информацию о товаре;
- Поля формы проверяются автоматически перед публикацией;
- В случае ошибок при публикации система выдает сообщения об ошибке с инструкциями как их с корректировать;
- После успешной публикации товар отображается на витрине в маркетплейсе.

Use cese: Публикация товара на витрине.
Актор: Продавец.
Предусловие: Продавец авторизован в системе и имеет права на добавление в товара на витрину.
1. Продавец инициирует процесс добавления нажав кнопку добавить товар на странице ЛК магазина, с товарами;
2. Система отображает форму добавления товара с полями для заполнения;
3. Продавец заполняет поля формы;
4. Продавец загружает не менее одной фотографии;
	4.1 Если при загрузке возникает ошибка (не верный формат или размер);
	4.2 Система выводит сообщение об ошибке и подсказку по ее устранению;
	4.3 Продавец загружают фотографию соответствующую требованиям;
5. Продавец нажимает кнопку добавить;
6. Система проверяет корректность заполнения формы;
	- обязательные поля заполнены;
	- содержимое полей соответствует требованиям (длинна, формат заполнения);
	- загружено не менее одной фотографии товара;
	6.1 Если на этапе проверки формы система фиксирует ошибку заполнения формы;
	6.2 Система выделяет не корректно заполненное поле(я) формы, и подписывает под ними подсказки по исправлению;
	6.3 Продавец вносит правки;
7. При успешной проверке система сохраняет данные товара в БД в статусе "ожидает публикации" (или промежуточном статусе  "ожидает модерации");
8. Система переключается на карточку добавленного товара и выводит уведомление об успешном добавлении;
9. Продавец нажимает кнопку опубликовать;
10. Система помечает товар как отображаемый на витрине.

Альтернативные сценарии:
	Иные сложности:
	1. На любом шаге возникает сложность технического характера (связь/доступ/работа отдельных компонентов системы);
 	2. Система выводит сообщение "Произошла техническая ошибка. Попробуйте еще раз позже или обратитесь в техническую поддержку";
	3. Система переводит Продавца на страницу ЛК магазина, с товарами.

Задача 3.
	3.1 По скриншотам формы и ошибок нужно описать REST api интерфейс для обработки данных передаваемых формой.
	Описание в формате OpenAPI: registration.yaml
	
	3.2 Алгоритм на стороне бекенда:
		1. Делаем POST-запрос на API reCAPTCHA.
			1.1 Если получаем ответ где значение параметра success не true, то выдаем сообщение об ошибке.
			1.2 Прерываем обработку запроса.
		2. Проверяем наличие в БД пользователя с user_name совпадающим с переданным user_name.
			2.1 Если совпадение найдено - выдаем сообщение об ошибке (5 слайд).
			2.2 Прерываем обработку запроса.
		3. Проверяем переданное значение полей password на валидность (содержит минимум один символ не относящийся к буквенно цифровой группе, содержит минимум один заглавный символ английского алфавита, содержит минимум один строчный символ английского алфавита, содержит минимум одну цифру, содержит минимум один спецсимвол, длинна пароля не менее восьми символов).
			3.1 Если поле не валидно, выводим сообщение об ошибке (4 слайд).
			3.2 Прерываем обработку запроса.
		5. Сохраняем БД (user_name, password, first_name, second_name).
